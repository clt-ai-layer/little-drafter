## 2. Analytical Summary (from DevInsights)

### 2.1 Key Logic & Implementation Details
- **Domain (Emmett Decider)**: The `Draft` aggregate will manage the lifecycle of a drawing session.
    - **State**: `TemplateId`, `CurrentStepIndex`, `Status` (`in-progress` | `completed`).
    - **Behavior**: `StartDraft`, `ProgressStep`, `UndoStep` (decrements index), `CompleteDraft`.
- **Event Sourcing**: Explicit `DraftStarted`, `DraftStepProgressed`, `DraftStepUndone`, and `DraftCompleted` events.
- **Content Delivery**: `CatalogService` serving `DrawingTemplate` objects. SVGs are pre-segmented with group IDs matching the step index.
- **UI Interaction**:
    - **Physical Affordance**: Large bottom nav bar with distinct "Next" (green/yellow pulse) and "Back" (red/orange) arrows.
    - **Visual Feedback**: Pulse effect (blue) on the latest revealed SVG path.
- **Audio Guidance**: Integration with a TTS provider to read metadata from SVG groups.
- **Projections**: `AchievementProjection` listening for `DraftCompleted` to update user stars/badges.

### 2.2 Key Scenario Groupings
- **Group 1: Navigation & Progression**: Validating the "Paper Support" model where progression is manual.
- **Group 2: Content Rendering & Cues**: Ensuring visual (pulse) and audio (TTS) cues align with the current step.
- **Group 3: Lifecycle & State**: Transitioning from start to completion and handling persistence/replay.
- **Group 4: Robustness & Edge Cases**: Handling missing assets, fast interaction, and boundary limits (min/max steps).

### 3.2 Task 2: Create 00.Scenarios.master.md
- **Analysis**: Map the 8 key decisions from DevInsights to specific scenarios.
- **Content**:
    - **Happy Path**: Start → Step 1 (Pulse/TTS) → Next → ... → Last Step → Next → Complete (Confetti).
    - **Edge Cases**:
        - Back from first step (no-op or reset?).
        - Next from last step (completion).
        - Multiple rapid "Next" clicks.
        - Resuming an `in-progress` draft.
    - **Error Handling**:
        - SVG load failure.
        - TTS engine timeout/failure (silent fallback?).
        - Event persistence failure.
- **Fit Matrix**: Validate that the Emmett Decider pattern perfectly fits the "back/forward" navigation requirements.

## 4. Report to Parent

### 4.1 Plan Coverage
This plan covers the transition from abstract decisions (Insights) to concrete implementation directives (Instructions) and validation cases (Scenarios). It ensures the "Paper Support" pivot is centrally documented.

### 4.2 Open Design Decisions / Blocking Questions
- **Back from First Step**: Should clicking "Back" on the first step do nothing, or show a "Quit" confirmation? (Assumption: No-op for simplicity in this MVP).
- **TTS Failure**: If TTS fails, should we show a text bubble, or just proceed with visual guidance only? (Assumption: Visual-only fallback).

### 4.3 Parent Relay to User
- The sub-agent is now generating the final master instructions and scenarios.
- The focus is on the "Paper Support" model where the app is a guide, not a tracing surface.
- We are assuming a silent fallback for TTS and a no-op for "Back" on the first step unless corrected.
